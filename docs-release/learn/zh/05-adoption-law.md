# 采用定律与完成的四层

团队构建的大多数东西并没有被真正采用。它发布了，通过了测试，有人演示了——然后旧方法又悄悄赢了回去，因为旧方法从来没有被真正拿掉。这一页要讲的，就是能解释这个现象的一条规则，以及我们由此推导出的*完成*定义。这些文档里的所有内容中，这是最值得记住的部分。

## 采用定律

> **一项能力被采用，当且仅当它是某条承重路径上唯一的合法路径。只要遗留方案仍然存在、迁移没有被强制执行，遗留方案就会凭惯性获胜。**

这就是定律的全部内容，而且它毫不留情。造出更好的工具还不够，让它可用也还不够。只要旧路径还能走，人们就会继续走——不是因为固执，而是因为惯性才是默认选项，切换总是有代价的。一个特性只有在它是*唯一*的通路，或者它要取代的那条路已经被真正拆掉之后，才算真正落地。

## 完成的四层

既然采用要求"唯一路径"，那么代码发布就不能算终点。因此我们把*完成*定义为四个递进的层次，每一层都必须成立：

1. **结构正义**。测试和 CI 全绿，没有硬门违规。这只是入场券——它证明代码能跑，证明不了它有意义。
2. **语义验收**。实现真正匹配它所声称要落实的决策——不是一个看似合理的相似物，而是那个决策本身。
3. **对抗式验证**。承重的部分要能扛住一次双盲红队测试。如果这东西重要，就该有人在不了解其内部构造的情况下试图攻破它。
4. **使用证明**。交付之后，真实使用出现——或者这东西本身就是一条强制路径，已经没有遗留替代方案还立着。

采用定律真正咬人的地方就在第四层。通过了前三层却在第四层失手的东西，叫作**已发布未使用（shipped-unused）**：它存在，能运行，但没人用它。已发布未使用不算完成的里程碑。当一次交付注定只能是骨架——比如基础设施本身还谈不上有用户——这种情况是允许的，但前提是它的接入工作要在同一时刻就排上日程，让它有一条通往第四层的既定路线，而不是一句遥遥无期的"以后再说"。

## 强化信号

这里有一个不起眼的推论，最后却成了最有用的一条经验法则。**我们无意识地反复用手工去够的那些结构，正是已经被行为验证过的创新。** 当你发现自己一遍又一遍手工重复同一件事时，这个习惯本身就是一场已经完成的实验在汇报结果——正确的反应是立刻把它并入系统。持续的、真实的手工使用，*就是*最强的使用证明；没有任何关于一个特性的说辞，能压过"人们不需要被要求就一直在用它"这个事实。

## 为什么这条规律能超越单个项目

以上这些都不专属于某一个工具。采用定律是人类和组织面对变化时的一条通用规律：一个更好的选项，只要还留着更差的选项可用，就会一直输给惯性，直到更差的选项被真正移除为止。无论你在构建什么——一个平台、一套工作流、一项标准、一个内部库——同样的检验标准都适用。如果它不是人们真正会走的那条路径上唯一的合法路径，那么无论测试多绿，都不能称之为"已采用"。这就是值得你从这些文档里带走、并用到下一个作品上的那一个想法。
