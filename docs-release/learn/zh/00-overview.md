# 这解决的是什么问题?

AI 代理(agent)极其擅长「做事」,却极其不擅长「记住自己为什么这么做」。给代理一个真实任务,它会在单次会话里生成一条密集的推理轨迹:为什么选这个方案而不是那个、做到哪一步了、沿途观察到了什么。然后会话一结束,这条轨迹几乎全部蒸发。

## 熵问题

每个会话都会产生三类持久内容,不管有没有人去捕捉它们:

- **选择(choices)** — "我们选 Postgres 而不是 SQLite,因为……"
- **进度(progress)** — "迁移做到一半了;索引重建卡住了。"
- **观察(observations)** — "10k QPS 下 p99 延迟是 50ms。"

默认情况下,这些内容都活在聊天记录里。也就是说:对话一旦翻篇,"为什么"没了,"进度"没了,"证据"也没了。下一个代理——或者一周后的你自己——只能从零开始。决策被悄悄地重新讨论一遍。工作被重做。整件事都无法审计,因为根本没有可审计的工件;有的只是一堵对话墙,没人会把它从头到尾完整重放一遍。

这就是熵:产生了却从未被消费的信息,不断堆积,直到和噪声再也分不清楚。代理用文字"记住"的东西越多,情况反而越糟——一个只进水不出水的池塘,迟早会变绿发臭。

## 我们的赌注:把这些内容做成一等公民,存进 git

我们赌的不是"做更好的笔记",而是把这条推理轨迹提升为**活在 git 里的结构化实体**——以 Markdown 作为真相的载体,代码库作为真相唯一的栖身之所。

这不是一块随手记的草稿板,而是一组有生命周期、有状态机、彼此之间存在类型化关系的实体。选择变成一个**决策(decision)**,可以被提议、被接受,日后也可以被推翻——并留下完整的记录。进度变成一个**任务(task)**,在既定的状态之间流转。观察变成一个**事实(fact)**,不可变,自带来源。因为它们只是版本控制下的纯文本文件,所以可以对比差异、可以审查、能长期留存,任何代理接手时都不需要先补一段热身对话。

## 三个主张

这篇概览提出三个主张,每个都有自己的一章展开。

1. **内核只有三个原语(primitive)**——decision、task 和 fact。其他一切都是这三者的组合。见 [01 · 三原语内核](01-three-primitive-kernel.md)。

2. **每个承重的写操作都要经过一扇门(gate),而门默认关闭(fail-closed)。** 证据不足或检查未通过,意味着这次写入会被**拒绝**,而不是被放行。见 [03 · 门与 fail-closed](03-gates-and-fail-closed.md)。

3. **一项能力只有成为唯一合法路径时,才算真正被采用。** 只要还有绕开它的办法,代理就会去走那条近路——所以这套机制必须**就是**那条路本身,而不是立在路边的一块提示牌(这就是采用律)。见 [05 · 采用律](05-adoption-law.md)。
