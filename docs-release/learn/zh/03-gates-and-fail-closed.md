# 门与 fail-closed

一条记录的可信度,取决于它被写入的那一刻。如果任何写操作都能不经检查就溜进仓库,整个结构——decisions、tasks、facts——就会退化成一堵散乱文字之墙,只是多了些仪式感。所以核心规则很简单,也很严格:

**每一次承重写入都必须经过一扇门,而门默认是关闭的。**

*fail-closed* 的意思就是字面意思。证据不足,或某项检查没通过,写入就会被**拒绝**,而不是被放行。安全的默认值是"否",你必须自己赚得那个"是"。默认开放的系统,信任的前提是"什么都没出错";fail-closed 的系统则假设"可能出错了",要求你先证明没出错,记录才会改变。

为了让这条规则真正可执行,承重写入不会直接落盘,而是统一经过一个 **写协调器(write coordinator)**——唯一的关卡,自动盖上水印并提交到 git。只有一道门,意味着只有一个地方需要落实规则,每一次被接受的写入都会留下持久、可追溯的痕迹。

## 三扇门

不同类型的写入,承重的方式也不同,所以一共有三扇门,各自守住一种失败。

| 门 | 防的是 | 触发条件 |
|---|---|---|
| **Exit Gate** | 一个里程碑其实并未完成 | 它的所有 decisions 都已解决,**且** task 链已闭合,**且** 事件账本完整 |
| **Usability Gate** | 一个能力存在却用不了 | 一个只配备了 `--help` 和能力清单的全新 agent,无法端到端跑通它 |
| **Disposition Guard** | 不安全的删除 | 有入边的东西要被移除——被引用的 decision 永不物理删除;fact 永不单独删除 |

**Exit Gate** 问的是一项工作是否真的完成了,而不是有没有人宣布它完成。宣布的成本很低;门检查的是结构。承重的 decisions 是否都已解决?task 链是否真正闭合,没有任何东西被阻塞或悬空?事件账本是否完整?只有全部满足,里程碑才算通过。

**Disposition Guard** 防止删除操作悄悄割裂关系图。任何有入边的东西都受保护:仍被其他实体指向的 decision 永远不会被物理删除——最多把它标记为退役,让它的历史和边保留下来。fact 永远不会被单独删除,因为可能有别的东西依赖它来追溯出处。你可以归档,但通常不能销毁。

## 为什么可用性也需要一扇门

Usability Gate 是三扇门里最不显眼的一个,它的存在源于一个惨痛的教训:**能力存在,不等于 agent 能用上它。** 一个功能可以被完整实现、完整测试,却在实践中完全触达不到——因为没有任何东西告诉 agent 它的存在,或者入口像迷宫一样绕,或者唯一能调用它的那条命令根本不出现在 `--help` 里。

一个 agent 找不到的能力,在实际意义上就等于没造出来。所以这道门把测试标准挑明了:一个**全新**的 agent,完全不记得这东西是怎么做出来的,只拿到自描述的表面信息(`--help`、能力清单),必须能靠这些信息把它跑通到一个可用的结果。如果做不到,这项工作就没完成——不管单元测试多绿都不算。这正是"搜索,不是记忆"这条原则的机械化落实:能力必须自己把自己"广而告之",不能指望有人记得它存在。

## 门是机制,不是标准

门是一套机械装置——它触发、它检查、它放行或拒绝。它不负责定义"完成"应该是什么意思,它只是执行别处定义好的标准。而那个标准——分层的完成定义,也就是 Exit Gate 及其同伴实际核对的对象——是另一个独立的话题:
